3. How it works
The Offline-First design works as follows:
1.
When a user sends a message, it is immediately written to the messages table and the outbox table.
2.
The UI observes the messages table via a Flow. Since Room is reactive, the message appears in the chat list instantly with a PENDING status.
3.
The outbox table acts as a persistent queue. Even if the app is killed or the phone reboots, the message remains there until the Sync logic (Phase 3) successfully sends it.
4. Android Fundamentals Used
•
Room Persistence: Used @Entity, @Dao, and @Database to create a local SQLite database.
•
Type Converters: Since SQLite doesn't support Enums, I added Converters to map Enums to Strings.
•
Repository Pattern: Decouples the UI from the data source. The UI doesn't care if data comes from a local DB or a remote server; it just talks to the Repository.
•
Reactive Streams (Flow): Used Kotlin Flow in DAOs to ensure the UI updates automatically when the database changes.
•
Hilt/Dagger: Automates "Dependency Injection," making it easy to swap a Mock Repository for a Real one later.
5. Edge Cases Handled
•
UUID for IDs: Using clientMessageId (UUID) instead of server-generated IDs. This allows messages to be created and referenced offline before the server even knows they exist.
•
Persistent Outbox: By making the outbox a database table (not just an in-memory list), we ensure no messages are lost on app crashes or reboots.

I have implemented the Login Screen with Background Video and the User Identity system. Here is the breakdown of the architecture, fundamentals, and justifications as requested.
1. Architecture & Approach
•
Identity System: I used Jetpack DataStore (Preferences).
◦
Why? It's the modern replacement for SharedPreferences. It's built on Coroutines and Flow, meaning all I/O is off the main thread, preventing UI stutters.
◦
Approach: A UserPreferencesRepository acts as the single source of truth for the user's name. This name is then injected into the MessageRepository to be used as the senderId.
•
Video Background: I used Media3 ExoPlayer.
◦
Immediate Playback: To avoid a blank screen, the player is initialized and prepare() is called within a remember block. This starts the buffering process the moment the LoginScreen enters the Composition.
◦
Memory Efficiency: To stay under 100MB RAM, I've configured the player to use a SurfaceView (via PlayerView), which is more memory-efficient than TextureView because it doesn't require the UI thread to manage the video frames.
•
Navigation: Used Jetpack Compose Navigation with a NavGraph that detects "First Launch" by observing the DataStore's login state.
2. Fundamentals Explained
Activity Lifecycle
The MainActivity is the "Host." It remains active throughout the app's life. However, we don't want the video player to keep consuming RAM when the user moves to the Chat screen.
Compose Lifecycle
We use DisposableEffect. This is a Composable that runs when it enters the "Composition" and provides an onDispose block that runs when it leaves.
•
Effect: As soon as the user logs in and the LoginScreen is swapped for the ChatScreen, exoPlayer.release() is called. This immediately frees up 40-60MB of RAM that the player was using.
Media Playback Lifecycle
Media playback involves complex states (Idle, Buffering, Ready, Ended). By calling prepare() and playWhenReady = true early, we minimize the time the user spends looking at a static screen. We also set repeatMode = Player.REPEAT_MODE_ALL for a seamless looping background.
3. Justification of Choices
•
DataStore over Auth API: As per your requirement, we need identity but not full Auth. DataStore provides a "production-grade" way to store this locally without the overhead of an Auth library.
•
Media3 over VideoView: VideoView is older and harder to optimize. Media3 (ExoPlayer) is the standard for high-performance apps and allows us to control the buffer size and rendering surface directly to meet your 100MB constraint.
•
SurfaceView over TextureView: SurfaceView has its own dedicated window layer, which means the GPU handles the video scaling and rendering directly, reducing the CPU and RAM load on the main application process.
